# CodeCache — Independent Project Review (2026-06-17)

**Reviewer role:** independent director-of-research / engineering audit.
**Scope:** the whole project — Rust crate (`codecache-rs`), Python research harness
(`research/r1_harness/`), and all documentation — verified for validity, correctness, and
reproducibility, with problems found and fixed.
**Method:** run every quality gate; validate the product end-to-end against a live repo; a
6-dimension multi-agent audit with adversarial verification of every finding; fix the confirmed,
low-risk issues; write user + research-verification docs.

---

## 1. Verdict

**CodeCache is in excellent shape and genuinely works.** Every quality gate is green, the product
runs end-to-end across all three v0.1 languages, the MCP server is a real (not stubbed)
implementation, and the research harness is methodologically careful and honest about its own
limits. The engineering quality is high: typed errors throughout, no reachable `unwrap/expect/panic`
in shipped code, deterministic ordering with explicit tie-breaks, and a hard token-budget ceiling.

The audit surfaced **32 confirmed findings, none critical**. They are overwhelmingly **documentation
drift** (stale test counts, an agent-facing tool description that overclaimed glob support, a wrong
field name in the MCP setup guide) plus **one real, latent code bug** (explicit re-index of an
unsupported file was silently treated as Python). All high-value, low-risk items were fixed in this
review; the gates remain green afterward (now **230 Rust tests**, +2 regression tests for the fixes
below, including the two pre-release follow-ups in §5).

## 3. Audit method

A deterministic multi-agent workflow audited six dimensions in parallel, then **adversarially
verified each finding** (an independent skeptic re-checked every claim against the actual code/doc
before it counted):

1. Scientific validity of the research harness
2. Research reproducibility & harness usability
3. Rust core correctness (parser/chunker/indexer/hasher)
4. Retrieval + CLI + MCP correctness
5. Documentation accuracy & internal consistency
6. Build / CI / release / packaging

**Outcome:** 32 confirmed findings (0 critical, 7 high, 16 medium, 9 low); **5 findings refuted** by
the adversarial pass (see §6). The verification step earned its keep — it killed several
plausible-but-wrong claims (e.g. a token-budget integer-overflow that is mathematically impossible at
realistic sizes).

## 5. Pre-release follow-ups — both completed (2026-06-17, TDD)

The two findings initially deferred were knocked out before the release, each test-first with gates
green:

1. **HF dataset revision pin** (`fetch_contextbench.py`, high/reproducibility) — **done.** Added a
   `--revision` flag (default `main`; pass a commit SHA for a fully-reproducible fetch) threaded into
   `load_dataset(revision=…)`, plus a sidecar `contextbench_verified_slice.meta.json` recording the
   dataset/config/split and the **requested + best-effort-resolved commit SHA**. The records-list
   cache keeps its bare-array shape, so the downstream `json.loads → list` loader is untouched. Pure
   `build_provenance()` helper, unit-tested (+2 research tests → **168 passed / 1 skipped**).
2. **Bare-`\r` (classic-Mac) line endings** in `parser::extend_to_line_end` (low) — **done.** Added a
   `Some(b'\r') => end + 1` arm (after the `\r\n` arm, so CRLF still consumes both bytes; only fires
   when `end` points at a `\r`, so a span is never over-extended). RED→GREEN unit test
   `extend_to_line_end_covers_lf_crlf_bare_cr_and_eof` (+1 Rust test → **230 passed / 0 failed**).

A handful of low-severity stylistic suggestions (e.g. a `debug_assert` for finite BM25 scores, minor
Windows path-separator consistency in one doc) were judged not worth the churn; the underlying code
is already correct.

## 6. Findings the adversarial pass refuted

The skeptic verifiers correctly rejected 5 findings — worth recording so they aren't "re-discovered":

- **Token-budget integer overflow** — mathematically impossible at realistic sizes (`len/4` token
  counts; overflowing `usize` would need exabytes of text).
- **`rust-version = "1.85"` vs toolchain `1.85.0`** — semantically identical to crates.io; not a defect.
- **Committed `runs/` artifacts may diverge from the harness** — `runs/` is gitignored and untracked;
  nothing is committed.
- **Cold-index 10K regression "undocumented"** — it is fully documented (ROADMAP D20, TODO, `benches/CLAUDE.md`).
- **Chunk-overlap dedup only tested for M4 chunks** — `retriever_tests.rs` covers crossing-overlap
  dedup with explicit byte spans.

## 7. Notable strengths (balanced assessment)

- **Honest research scope discipline.** The harness never overclaims: every run entrypoint prints a
  scope/honesty disclaimer, R2.2's headline is a correctly-framed *null result*, and all arm-winner
  claims are deferred to the (gated) R3. A team caught and fixed its own double-indexing measurement
  bug (D29) in review — exactly the discipline you want.
- **Two-implementation metric cross-check.** Layer-1 metrics exist in Rust (`retrieval_quality.rs`)
  and Python (`scorer.py`), with mirrored hand-computed tests — a strong guard against silent metric
  bugs.
- **Determinism and safety as first-class concerns.** Stable tie-breaks `(bm25, file, start, end)`,
  `f64::total_cmp` to avoid NaN panics, parameterized FTS5 (no injection), a hard token ceiling, and
  graceful degradation so indexing never hard-fails on one malformed file.
- **Documentation depth.** Per-module `CLAUDE.md` files, a Decision Log (D1–D31), and milestone phase
  plans make the codebase unusually auditable. The drift this review fixed was almost entirely stale
  *counts/labels*, not wrong *architecture*.

## 8. Bottom line

The project is **valid, working, and well-engineered.** After this review it is also more internally
consistent (doc drift removed), more robust (the unsupported-file footgun closed and tested), and
better documented for both users (`USAGE.md`) and research reviewers
(`docs/RESEARCH_VERIFICATION.md`). All gates are green. The natural next milestones are the ones the
roadmap already names: the gated **R3** agent-in-loop study and the **R4** write-up/release.
